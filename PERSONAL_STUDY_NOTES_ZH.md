# ESP32 股票小屏个人学习笔记

## 1. 这个项目本质上在做什么

这个项目本质上不是“把股票价格显示出来”这么简单。

如果只看最表层，它是：

- 用 ESP32 小屏显示美股价格

但如果从产品和工程角度看，它其实是在练这几件事：

- 小硬件产品的启动流程设计
- 设备联网失败后的恢复机制
- 嵌入式 UI 的信息布局
- ESP32 上网络请求稳定性
- 用代理服务降低设备端复杂度
- 让一个 demo 逐步长成“可复用的小产品”

所以这个项目很适合反复研究，因为它不是单点知识，而是很多能力揉在一起。

---

## 2. 这块板子的特点

用的是：

- Waveshare ESP32-S3-LCD-1.47B

这块板子对这个项目特别合适的地方在于：

- 自带屏幕
- 自带按键
- 自带 RGB 灯
- 有 WiFi
- 小巧，适合做桌面摆件或者状态屏

但它也有明显限制：

- 内存和网络资源有限
- 直接访问公网 HTTPS 容易不稳定
- 多任务并发时比较容易踩坑
- 设备一旦进入异常状态，排查比 PC 程序难很多

---

## 3. 这个项目最后形成的结构

我现在可以把它理解成 5 层：

### 第 1 层：显示层

负责：

- 启动画面
- WiFi 扫描画面
- setup mode 画面
- 股票行情页
- 公司信息页

对应文件：

- `LVGL_Example.cpp`

### 第 2 层：流程层

负责：

- 开机先做什么
- 连上 WiFi 后做什么
- 连不上 WiFi 后做什么
- 什么时候进入 setup mode

对应文件：

- `LVGL_Arduino.ino`

### 第 3 层：配置层

负责保存：

- WiFi
- 股票代码
- 代理地址
- 亮度
- 刷新时间

对应文件：

- `AppConfig.cpp`

### 第 4 层：联网层

负责：

- 扫描 WiFi
- 连接保存过的 WiFi
- 时间同步

对应文件：

- `Wireless.cpp`

### 第 5 层：数据层

负责：

- 请求股票数据
- 管理当前股票
- 自动刷新
- 保留上次成功价格

对应文件：

- `Stock.cpp`

## 4. 为什么最后要做 setup mode

这是这个项目里非常关键的一步。

一开始最麻烦的事情其实不是显示股票，而是：

- 每次换 WiFi 都要改代码
- 每次改代理地址都要重新编译
- 去了新环境就很难用

所以 setup mode 的价值是：

- 让设备自己开热点
- 手机连上后直接网页配置
- 不需要每次为了改 WiFi 去改源码

这一步让项目从“实验代码”变成了“可复用设备”。

这也是以后做任何带联网的小硬件时都值得记住的思路：

> 用户环境会变，配置必须脱离源码。

---

## 5. 为什么要用代理，而不是 ESP32 直接访问股票 API

这是另一个核心经验。

一开始会很自然地想：

- ESP32 连网
- 直接请求 API
- 把结果显示出来

但实际做下来会发现：

- HTTPS 对小设备压力大
- 重复请求容易不稳定
- 网络错误排查复杂
- 不同 API 的限制和兼容性很多

所以最后采用了：

- ESP32 -> 代理
- 代理 -> 股票 API

这带来的好处是：

- ESP32 只需要处理更简单的 HTTP
- 可以统一数据格式
- 可以做缓存
- 可以换数据源而不一定改设备端很多代码

这是很典型的“让弱设备做轻事，让强设备做重事”的架构思路。

## 6. 这个项目里踩过的几个大坑

### 坑 1：编译问题很多不是代码错，而是环境错

比如：

- `lvgl.h` 找不到
- `SD_Card.h` 找不到
- sketch 太大
- Arduino 缓存损坏

这个项目让我学到一点：

> 嵌入式开发时，先确认环境、板卡配置、库版本，再怀疑自己代码。

### 坑 2：显示问题不一定是 UI 代码问题

例如：

- 横竖屏不对
- 画面镜像
- 启动花屏
- 背光闪烁

很多时候根因不在 LVGL 本身，而在：

- LCD 初始化顺序
- 背光时序
- PWM 频率
- 刷屏实现方式

### 坑 3：小设备的“多任务并发”很容易出事故

这个项目里我们碰到过：

- task 栈溢出
- WiFi 流程互相打架
- TCP/IP core 断言崩溃

最后得到的经验很实用：

> 在 ESP32 这种设备上，流程简单、网络单线程、任务职责清晰，往往比“写得很灵活”更重要。

### 坑 4：能显示不代表算完成

早期虽然能显示股票价格，但实际上：

- 启动不稳
- 换 WiFi 麻烦
- 网络失败恢复能力差

后面真正做成“可以一直用”的关键，不是加更多功能，而是：

- 稳定启动
- 正确恢复
- 可配置

## 7. 当前版本最值得记住的设计

### 启动流程

现在设备启动是：

1. 启动画面
2. WiFi 扫描和尝试连接
3. 成功则进股票页
4. 失败则进 setup mode

这个流程非常像一个真正的小设备，而不是一段程序。

### 最后成功的关键

这几个改动是非常关键的：

- 提高 Driver task 栈大小
- 修掉 LCD 写屏时的大栈缓冲
- 去掉重复的 WiFi 重连流程
- 避免多个任务同时碰网络栈
- 提高背光 PWM 频率
- 把配置从源码中抽出来

这些都不是“炫功能”，但对稳定性价值最大。

这个项目后面完全还能继续长。

我觉得最值得做的方向有：

### 方向 1：继续做成产品

- 更完整的配置页
- 股票排序 / 增删
- 更好看的 setup 页面
- 外壳
- 开机品牌感

### 方向 2：继续练嵌入式工程能力

- 减少阻塞
- 更稳的重连机制
- 更好的内存管理
- 更完整的异常日志

### 方向 3：继续练产品经理 + 硬件结合能力

这个方向其实很适合你。

因为它不是纯电子，也不是纯代码，而是：

- 一个具体用户场景
- 一块真实硬件
- 一个可感知的最终产品

这个过程特别适合重新进入硬件世界。
